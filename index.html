This needs a session-claim system: each client generates a random session ID, writes it to the account's Firestore doc when it starts playing, and listens for changes. Clean up on sign-out In the $('fbOut').onclick handler, add before the signOut: jsif(sessUnsub){sessUnsub();sessUnsub=null;} How it behaves Device A is playing → Device B signs in and claims → A's onSnapshot fires within a second or two, freezes the game, and drops A to the login screen with the takeover message. "Last login wins" is the right model here (rather than blocking B), because otherwise a crashed or closed tab would lock the account out forever — with this design, recovery is just signing in again. Two honest caveats: It's client-side enforcement. A modified client can simply ignore the kick. For your casual leaderboard game it's plenty, but if you want it hard-enforced, the follow-up is a Firestore security rule requiring writes to include the current activeSession — that stops even modified clients from saving from a stale session. Guest mode is untouched — guests are local-only with no account to protect, so nothing to abuse there. Also note the kick screen intentionally doesn't call FB.auth.signOut() — the auth session stays valid, so the kicked player can immediately tap Sign in to steal the session back. If two people share an account they'll ping-pong kick each other, which is exactly the deterrent you want.
